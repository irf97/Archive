The Four Specs You've Written

1. ARGOS Operational Specification

  • What it is: The memory substrate specification
  • Runtime: Phoenix server on 127.0.0.1:4020, Postgres source-of-truth
  • Contract: One desktop daemon, many MCP + HTTP clients
  • Core principle: Capture fast, slow work happens after
  • Load-bearing invariant: No hidden autonomy — canon changes require operator approval
  • Current state: Implemented through Phase 8 (event log, canons, approvals, proposals, scoring stubs)

2. Argos Automated Research Spec

  • What it is: The research workflow specification for turning chats into durable knowledge
  • Flow: Chat intake → extraction → research job → ingest → review → outcomes
  • Core rule: No unsourced claims in memory; all claims need sources before promotion
  • Objects: chatIntake, hypothesis, source, claim, question, decision, domain, outcome
  • Quality gates: No source → needs_source; contradiction → disputed; task without evidence → blocked
  • Current state: Specification only; ready for implementation

3. Convergent Cell Program Specification (DRAFT)

  • What it is: The normative specification of how the three systems interoperate
  • Scope: Event schema, identity/authority/trust, architecture profiles (M/P/B/H), transport, conformance
  • Status: DRAFT — honest about gaps; open items documented
  • Conformance: Levels 0–3; executable acceptance suite in §8
  • Purpose: Make the program buildable from zero; guard against "vision with no wires"
  • Current state: Specification only; ratification pending

4. ENS Navigator Product Spec

  • What it is: The research operations app specification
  • Purpose: Expert discovery, domain knowledge curation, outcome execution
  • Pages: Navigator, Expert Radar, Domain Knowledge, Outcomes, Ingest Queue, Spec Pages
  • Data model: Experts, pipeline, domain knowledge, outcomes, crawl jobs, ingest log, saved views
  • Local API: Read/write endpoints for expert, domain knowledge, outcomes, crawl ingestion
  • Quality gates: Source validation, contradiction detection, evidence gating on tasks
  • Current state: Implementation in progress on localhost:8787

How They Actually Work Together

Flow 1: Knowledge Creation (Chat → Memory → Execution)

1. You chat with Codex in Claude Code
   └─ Session produces decisions, code, insights, questions

2. Codex calls /api/capture → ARGOS event log
   └─ Raw event: kind=codex_session, content, decision_ids

3. Chatdrift ingestor reads Claude Code transcript daily
   └─ Detects drift: "this conversation started about X, ended about Y"
   └─ Emits: kind=drift_detected, scope=software, origin_intent, terminal_state

4. ARGOS intelligence loop processes drift event
   └─ Surfaces proposal: "should we add entry to constellation?"
   └─ Surfaces proposal: "should we create research task?"

5. You review in dashboard (Mission Control)
   └─ Approve drift proposal
   └─ Approve "create research task" proposal

6. Research task spawns research job
   └─ Job runs: "find sources supporting this claim"
   └─ Results ingest into ENS Navigator Domain Knowledge

7. ENS Navigator "Outcomes" creates task graph
   └─ Task depends on sources
   └─ Task has evidence gates

8. Task completion triggers outcome scoring in ARGOS
   └─ Score conditions on operator state at task creation
   └─ High-signal outcomes bias next context pack

9. Next Codex session receives better context pack
   └─ Outcome-proven claims rank higher
   └─ Scope-relevant knowledge prioritized
   └─ Loop closes

Flow 2: Skill Evolution (Invocation → Ranking → Use)

1. You use a skill during Codex session
   └─ Codex calls argos_invoke_skill(name, context)
   └─ Constellation logs: invocation_count++

2. Constellation dashboard shows invocation heat map
   └─ High-invoked skills bubble to top
   └─ Unused skills eventually demoted

3. Next time you need context, pack compiler reads Constellation
   └─ Highest-invoked skills default into pack
   └─ Skill ranking is explicit + visible

4. Outcomes from sessions using those skills feed adaptive loop
   └─ Good outcome + high-ranked skill = skill stays ranked
   └─ Bad outcome + high-ranked skill = skill gets downranked

5. Constellation auto-suggests new entries when drift detected
   └─ Chatdrift says "new architecture pattern detected"
   └─ Claude API drafts constellation entry
   └─ You confirm or reject

Flow 3: Research Validation (Unsourced → Sourced → Outcome)

1. Chat intake captures raw conversation
   └─ Stored as chatIntake object
   └─ Marked "never approved directly"

2. Extraction splits chat into claims, questions, hypotheses
   └─ Claim: "IPv6 CGA binds key to address"
   └─ Question: "how does CGA key rotation work?"
   └─ Hypothesis: "Football United should use CGA for identity"

3. Research job created: "find sources for IPv6 CGA claim"
   └─ Codex crawls papers, RFCs, official docs
   └─ Results posted to /api/domain-knowledge/ingest

4. ENS Navigator review lanes show what needs sourcing
   └─ Move claim from "Inbox" → "Needs Source" → "Approved"
   └─ Or "Disputed" if contradicts existing claim

5. Approved claims feed Outcomes
   └─ Create task: "draft IPv6 UDP multicast packet header"
   └─ Task depends on: claim about CGA, decision about multicast
   └─ Task has evidence gates: must reference sources

6. Task completion flows back to ARGOS outcome scoring
   └─ High-quality research → high-signal outcome
   └─ Next research job biased toward similar patterns

The Key Architectural Decisions

1. Local-First, Not Remote

  • All data on local machine
  • One Phoenix process
  • Postgres on localhost
  • No cloud sync (yet)
  • Means: your research is yours; no privacy boundary

2. Approve-Everything Default (with autonomy escape hatch)

  • ARGOS INV-2: canon changes require approval by default
  • v1.2 spec allows per-canon autonomy_policy declarations
  • Low-stakes updates can auto-apply with audit trail
  • High-stakes always wait in queue
  • Means: you stay in control; speed comes from delegated authority, not hidden autonomy

3. Sourcing Before Memory Promotion

  • Raw chat never becomes approved knowledge
  • All claims must carry sourceIds
  • Disputes surface as remediation tasks, not disappearances
  • Rejected knowledge can still spawn pressure-test outcomes
  • Means: memory is auditable; you can trace where every claim came from

4. Explicit Outcome Dependencies

  • Tasks need evidence (sources or claims) before moving to Done
  • Dependencies block tasks until resolved
  • Evidence gates prevent incomplete outcomes
  • Remediation tasks for disputed knowledge
  • Means: you can't accidentally build on sand

5. Signal Over Autonomy

  • Drift detection is visible, not automatic
  • Skill ranking is visible, not automatic
  • Proposals surface to queue, not auto-apply (except bounded autonomy)
  • Dashboard is the inbox
  • Means: you can't miss what's important, and you control the pace

Integration Points (How Data Flows Between Systems)

ARGOS ↔ Constellation

  • ARGOS reads: skill entries via HTTP GET :7777/state (daily poll)
  • Constellation reads: operator state via ARGOS API
  • Data: skill invocations, usage ranking, entry metadata
  • Event: When high-invocation skill found, feed to next context pack

ARGOS ↔ Chatdrift

  • Chatdrift reads: ARGOS canons to check for architecture drift
  • ARGOS reads: drift detection results from Chatdrift
  • Data: chat transcripts, drift vectors, origin→terminal deltas
  • Event: When drift detected, create proposal in ARGOS queue

ARGOS ↔ ENS Navigator

  • ENS Navigator reads: operator approval decisions from ARGOS queue
  • ARGOS reads: outcome tasks + evidence from ENS Navigator
  • Data: domain knowledge, claims, sources, outcomes
  • Event: When outcome task done, trigger scoring in ARGOS

ENS Navigator ↔ Constellation

  • Constellation reads: expert/source recommendations from domain knowledge
  • ENS Navigator reads: skill recommendations for research tasks
  • Data: people, sources, research jobs, skill invocations
  • Event: When new expertise pattern found, suggest constellation entry

All → NATS event bus

  • Published: drift_detected, outcome_scored, skill_invoked, proposal_surfaced, canon_changed
  • Subscribers: ARGOS (proposals), Chatdrift (validation), Mission Control (dashboard)
  • Purpose: Real-time sync without polling; allows async work

All → Mission Control Dashboard

  • Data sources: ARGOS events, Chatdrift drift vectors, Constellation skill ranking, ENS Navigator outcome graphs
  • Display: Unified view of substrate state
  • Interaction: Approve proposals, review outcomes, monitor system health

What Makes This Different From Typical Research Apps

Not a summarizer

  • Doesn't compress conversations
  • Preserves explicit source linkage
  • Tracks drift (origin ≠ terminal)
  • Prevents memory collapse

Not an autonomous agent

  • No hidden decisions
  • Approval-gated by default
  • Operator controls approval rate
  • Audit trail for everything

Not a knowledge graph

  • Not a semantic web
  • Not a belief/confidence system
  • Evidence-gated (sources required)
  • Quality-gated (contradictions surface, not hidden)

Not a task manager

  • Task dependencies explicit
  • Evidence gates prevent incomplete work
  • Outcomes link to claims + sources
  • Rejected knowledge becomes pressure tests

Not a chat exporter

  • Chat never becomes memory directly
  • Requires extraction → research → review
  • Three-gate system: source, review, outcome
  • Human always in the loop

  2. Check Constellation skill ranking → reassign high-value skills
  3. Review "Disputed" claims in ENS Navigator → resolve or remediate

Monthly

  1. Full substrate review: what emerged? what shifted?
  2. Update ARGOS canons based on outcomes
  3. Suggest new Constellation entries based on drift patterns

Success Criteria

You'll know this is working when:

  1. Drift is visible — Chatdrift shows you've shifted architecture every 2 weeks, and it's on the dashboard
  2. Skills compound — Your top-5 skills at month-end are measurably more useful than month-start
  3. Research biases — Next month's research jobs bias toward domains where you had good outcomes
  4. Context improves — Codex context packs get measurably more relevant over time
  5. Approval rate sustains — You can review and approve the queue in <30 min/day without falling behind
  6. Memory is auditable — You can trace any claim to its source and see exactly why it's approved/disputed

If any of these isn't happening, the system is telling you something is broken or misconfigured.

The One Thing Everything Depends On

The approval queue must not overflow.

If proposals back up faster than you can review, the system collapses into silent autonomy (you give up checking) or total paralysis (you stop working).

The v1.2 bounded autonomy mechanism (INV-2) and the rate-limiting in ARGOS (IL-3) exist solely to prevent this.

If you find yourself with 100+ unapproved proposals sitting for more than 24 hours, pause and recalibrate:

  • Lower surfacing rate (IL-3 bounds)
  • Enable more autonomy_policy modes (INV-2)
  • Or reduce research job volume

The queue depth is the leading indicator of substrate health.
