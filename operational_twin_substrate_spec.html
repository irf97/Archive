Digital twin substrate

Robodex acts against a governed operational twin, not raw intention.

The missing layer between research/governance and physical execution is a living site model: physical reality, hardware reality, software reality, workflow reality, and authority reality represented as convergent, auditable state.

02

The five realities inside the twin

Physical

Zones, routes, dimensions, obstacles, fixtures, docks, geometry, affordances.

Hardware

Robots, sensors, devices, capabilities, limits, batteries, camera rigs, safety state.

Software

Services, APIs, queues, databases, credentials, permissions, integration endpoints.

Workflow

Tasks, handoffs, procedures, service flows, dependencies, evidence gates.

Authority

Roles, approvals, ownership, policy, escalation paths, autonomy boundaries.

Runtime

Telemetry, active missions, alerts, proofs, refusals, divergences, current projections.

06

What ARGOS must govern

Approval boundary

No sourced claim, workflow, or autonomy rule becomes accepted twin truth without ARGOS policy.

Memory lineage

Every twin projection points back to events, sources, operator approvals, and factset hashes.

Operator state

Risky approvals can depend on fatigue, mode, or current operating context.

Autonomy policy

Low-risk updates may auto-apply; high-risk changes wait in queue.

08

Event-first twin updates

The twin should be replayable. Current state is a projection over approved assumptions, observed facts, telemetry, and runtime proofs.

Event types:
  ens.claim.approved
  argos.policy.approved
  twin.zone.created
  twin.device.registered
  twin.workflow.approved
  robodex.cartridge.forged
  robodex.mission.refused
  robodex.task.completed
  telemetry.device.observed
  twin.divergence.detected

Projection:
  project(events, view="operational_twin")
    → { state, factset_hash, evidence_refs, warnings }
09

Build order

  1. Define operational_twin.v0 bridge schema.
  2. Map it deterministically to WorldModel.v1, HardwareSpec.v1, and CartridgeForgeRequest.v1.
  3. Add ENS domain-template ingestion into ARGOS approval queue.
  4. Implement twin projection from approved events.
  5. Generate Robodex forge inputs from twin projection.
  6. Feed Robodex proof/refusal events back into ARGOS and the twin.
  7. Add Mission Control view: twin health, stale facts, unresolved divergences.
10

Hard guardrails

No direct chat-to-robot path. Human or aen intent becomes a proposed event, not actuator authority.
GuardrailReason
ENS cannot approve itselfResearch evidence is not accepted truth until ARGOS policy allows it.
ARGOS cannot fake physical realityApproved assumptions must still be checked against telemetry and Robodex refusals.
Twin cannot executeIt represents reality; Robodex executes/refuses.
Robodex cannot rewrite canonIt emits proof/refusal evidence; ARGOS governs memory promotion.
aen cannot bypass convergeStructured language is proposer input, not hidden authority.
